# 动态菜单路由 + 图标管理 + 用户管理 设计方案

日期：2026-07-14
状态：已确认，待实施

## 一、背景与现状

### 1.1 租户 ↔ 角色 ↔ 菜单 ↔ 按钮 的既有关联

租户**不直接关联**菜单和按钮。`sys_menu` / `sys_button` / `sys_menu_btn` 三张表均无 `tenant_id`、均不实现 `ITenantEntity`，它们是**平台级全局字典**，所有租户共用一套菜单目录。

租户通过**角色**渗入权限体系：

| 表 | tenant_id | 说明 |
|---|---|---|
| `sys_user` (`Sysuser.cs:104`) | ✅ | 用户所属租户；`role_id` (`:24`) **单角色直挂**，无多角色关联表 |
| `sys_role` (`Sysrole.cs:19`) | ✅ | `0` = 平台共享角色（超管维护），非 0 = 租户自建角色 |
| `sys_role_menu_btn` (`SysroleMenuBtn.cs:20`) | ✅ | **角色-菜单-按钮三元授权表**，随角色隔离；`button_id=0` 表示仅授菜单 |
| `sys_menu` / `sys_button` / `sys_menu_btn` | ❌ | 平台全局字典 |

完整链路：

```
用户 --tenant_id--> 租户
用户 --role_id----> 角色 --sys_role_menu_btn--> 授权菜单 + 授权按钮
```

超管判定（`LoginPublicMode.cs:51`）：角色须为**平台级根角色**（`tenant_id == 0 && parent_id == 0`）。租户自建的顶级角色不算，防越权。

### 1.2 已确认的缺口

1. **前端缺用户管理页**。后端 `SysuserController` 端点齐全，前端 `api/system.ts` 里 `queryUser` / `insertUser` / `updateUser` / `postSetUnit` / `getUserRoleList` 等十余个封装**全是死代码，零处调用**。`views/system/` 下只有 `unit` / `menu` / `role` / `button`，**没有 `user/`** → 无任何界面能把用户挂到租户和角色上，权限闭环断裂。

2. **菜单图标形同虚设**。`menu/form.vue:94` 是纯文本输入框；仓库里的 `ReIcon/src/Select.vue`（IconSelect）**从未被任何页面引用**；图标格式混乱（`ri/settings-3-line` 斜杠 vs `ep:cpu` 冒号）。

3. **动态路由从未通电**。`router/utils.ts:261-268` 的 `initRouter()` 直接 `handleAsyncRoutes([])`。经 git 追溯，该注释自 `8e32b83`（vuefrontend 整体迁入）第一天就存在 —— 是上游模板原始状态，**不是踩坑后关闭的**。

4. **菜单表装的是幽灵数据**。库中 15 条菜单，除 `IotDevice` 外全部指向不存在的页面（`/record/alarmLog/index`、`/environmentalMonitor/environmentalMap/index` 等），且 `MenuCode` 与前端路由 `name` 对不上（`System` vs `SystemManage`、`buttonManagement` vs `SystemButton`）。**这极可能就是动态路由被注释的真实原因** —— 一开就全线 404。

## 二、决策

| # | 决策 | 理由 |
|---|---|---|
| 1 | **恢复后端动态路由**，菜单表成为侧边栏唯一真源 | 用户明确选择（C 方案） |
| 2 | `sys_menu` **加 `component` + `meta_json` 两列**，不加 `redirect` | 显式配置优于 path 约定匹配（后者有静默错配风险）；`redirect` 全部等于第一个子级，`addAsyncRoutes:403` 自动处理 → YAGNI |
| 3 | **删除** `iot.ts` / `system.ts` / `scada.ts` / `report.ts` 四个静态路由模块；**保留** `home.ts` / `remaining.ts` / `error.ts` | 唯一真源（DRY）。`home.ts` 提供 `path:"/"` 根路由 —— `formatTwoStageRoutes` 依赖它，且 `handleAsyncRoutes` 往 `router.options.routes[0].children` 挂载动态路由，**删了动态路由无处可挂** |
| 4 | `remaining.ts` 埋一条 `showLink:false` 的 `/system/menu` **逃生舱** | 菜单数据写坏时，超管手敲 URL 仍能进菜单管理页自救 |
| 5 | **删除** `filterNoPermissionTree` / `pruneByGranted` / `granted-menu-names` 静态裁剪 | 后端 `GetMenuTree(islimit=1)` 本就只下发授权菜单，前端再裁一遍是重复真源 |

## 三、必须修复的既有缺陷（否则动态路由一开即崩）

### 缺陷 A：`meta.rank` 下发的是字符串
`SysmenuController.cs:308` — `rank = item.SortBorder`，值形如 `"A002"`。
前端 `ascending()` 执行 `a.meta.rank - b.meta.rank` → **NaN，顶级菜单排序完全失效**。
**修**：`MetaInfo.rank` 改为 `int?`，由 `SortBorder` 去掉首字母后的数字部分解析（`"A002"` → `2`）。仅顶级菜单需要 rank。

### 缺陷 B：`islimit==1` 不补全祖先节点
`SysmenuController.cs:252-278` 只取 `sys_role_menu_btn` 中直接授权的 `menu_id`。
若角色只勾了子菜单未勾父目录，`GetMenuInfoList` 从 `"0"` 递归时找不到父节点 → **整棵子树静默消失**。
**修**：取到授权 menu_id 后，沿 `ParentId` **递归补全全部祖先节点**（参考 AiNet `AiAuthController.cs:46-52`）。

### 缺陷 C：`InsertUser` 把用户租户写死为操作者租户
`SysuserController.cs:89` — `info.TenantId = optmdl.TenantId`。超管的 `TenantId` 为 0 → **超管建的用户全部落到 tenant_id=0**，无法为其他租户建用户。
**修**：超管可指定 `TenantId`；非超管强制本租户。

### 缺陷 D：`GetRoleList` 不按目标租户过滤
`SysuserController.cs:398` 按角色层级返回可分配角色，但超管为租户 A 建用户时，应只能选**租户 A 的角色或平台共享角色（tenant_id=0）**。
**修**：`GetRoleList` 增加 `tenantId` 参数，按目标租户 + 平台共享过滤。

## 四、模块设计

### 模块一 · 菜单表扩容与种子重建（后端）

**DDL**（`sys_menu` 加两列）：

| 列 | 类型 | 用途 |
|---|---|---|
| `component` | varchar(200) 可空 | Vue 组件相对路径，如 `iot/center/index.vue`。**目录节点留空** |
| `meta_json` | varchar(500) 可空 | 自定义 meta 的 JSON，如 `{"projectKind":"scada"}` |

`meta_json` 的存在是硬需求：`scada.ts` 的 `/project` 与 `report.ts` 的 `/report/project` **component 完全相同**（都是 `@/views/project/index.vue`），仅靠 `meta.projectKind: "scada" | "dash"` 区分。若菜单表只存 component，组态与报表两个入口会塌成同一个页面。

**DTO**：`MenuInfo` 增 `component` 字段；`meta_json` 反序列化后 **merge 进 `meta`**。

**种子重建**：清空全部 15 条幽灵菜单，按下表重建 25 条。

### 模块二 · 图标真正可用（前端，移植 AiNet）

- 移植 `build/gen-ri-iconset.mjs` → 生成 `public/icons/ri.json`（全量 Remix Icon 3100+，约 987KB，**同源加载，不走 CDN**）
- 移植 `ReIcon/src/riCollection.ts`（`addCollection` 运行时一次性注册，幂等 Promise）
- `layout/index.vue` 启动时调用 `ensureRiCollection()`（非阻塞）
- 移植 `components/IconSelect/index.vue`（弹窗 + 搜索 + 滚动增量加载 PAGE_SIZE=120）
- **删除** `ReIcon/src/Select.vue`（从未通电的在线版空壳）
- `iconifyIconOffline.ts` 打补丁：`@iconify/vue/dist/offline` → `@iconify/vue`（已注册图标走本地注册表，未注册的才 CDN fallback）
- `package.json` 加 `gen:icons` / `predev` / `prebuild` 钩子 + devDep `@iconify/json`
- 菜单管理页：图标字段 `el-input` → `<IconSelect>`；列表列渲染**真图标预览**
- **统一图标格式为 iconify 冒号写法**（`ri:xxx-line`），归正现存的斜杠格式

### 模块三 · 动态路由通电（前端）

- `initRouter()` 恢复调用 `getAsyncRoutes(1)`，删除 `handleAsyncRoutes([])` 短路
- 删除 `router/modules/` 下 `iot.ts` / `system.ts` / `scada.ts` / `report.ts`
- `remaining.ts` 增加逃生舱路由 `/system/menu`（`showLink: false`）
- 删除 `filterNoPermissionTree` / `pruneByGranted` / `granted-menu-names` 静态裁剪机制
- 验证 `addAsyncRoutes` 对**目录节点**（component 为空）的处理：pure-admin 将动态路由拍平后全部挂到 `/` 的 children 下，目录节点仅用于菜单树渲染。此处**必须实测**，不能纸上推演。

### 模块四 · 用户管理页（前端新建 + 后端修 C/D）

新建 `views/system/user/`（列表 + 新建/编辑 + 重置密码 + 启停用），接通 `api/system.ts` 里的死代码封装。

表单字段（参考 AiNet `views/auth/account`）：

- 新建：**所属租户**（仅超管可见可选）· 账号 · 密码 · **角色**（按所选租户过滤）· 姓名 · 性别 · 手机 · 备注
- 编辑：账号只读 · 角色 · 启用状态 · 姓名 · 性别 · 手机 · 备注（不改密码）
- 重置密码：独立弹窗
- 列表列：账号 · 姓名 · 所属租户（仅超管）· 角色 · 手机 · 状态 · 最后登录 · 操作

后端配套修复 **缺陷 C**（`InsertUser` 租户）与 **缺陷 D**（`GetRoleList` 按租户过滤）。

## 五、实施顺序（风险后置）

1. **模块四** — 用户管理页 + 后端缺陷 C/D。独立，静态路由下即可验证，先补上权限闭环。
   （此阶段先在 `system.ts` 临时加一条 `/system/user` 静态路由，模块三时随文件一起删除。）
2. **模块二** — 图标。独立，不依赖动态路由。
3. **模块一 + 三** — 菜单表扩容 + 种子重建 + 动态路由通电 + 缺陷 A/B。一次性、高风险，前两块验证过后再动。

## 七、实施记录与相对原设计的偏差

实施于 2026-07-15 完成。三处必要偏差：

1. **`home.ts` 必须保留**（原设计写的是"业务路由全删"）。
   `router/index.ts:55` 的 `formatTwoStageRoutes` 要求存在 `path: "/"` 的根路由，且 `handleAsyncRoutes` 把动态路由挂到 `router.options.routes[0].children`。
   删掉 `home.ts` 会让动态路由无处挂载。实际删除的是 `iot.ts` / `system.ts` / `scada.ts` / `report.ts` 四个。

2. **逃生舱路径改为 `/rescue/menu`**（原设计写的是 `/system/menu`）。
   用 `/system/menu` 会与数据库里 `SystemMenu` 菜单的动态路由撞 `path` —— 静态路由先注册会抢占匹配，导致侧边栏高亮与标签页错乱。

3. **`sys_menu_btn` 按钮绑定重映射保留**（原设计写的是清空）。
   旧绑定里有 4 个菜单（菜单/按钮/角色/用户管理）在新菜单表里仍有对应节点，按新 `menu_id` 重映射即可保留 19 条绑定，无需重配。

另外顺带修掉的既有缺陷（页面上线后会被真实触达）：
- `EnableUser` 在 `info == null` 时访问 `info.UserUid` 的空引用崩溃
- `DeleteUser` / `EnableUser` 可以删除或禁用当前登录账号（补了自我保护）
- `UpdateUser` 完全无校验即可改任意用户的 `RoleId`（补了自我提权防护与角色归属校验）

**已知遗留**：`PostResetPwd` 把密码重置为硬编码的 `"new12356"`（`SysuserController.cs`），弱且不可配置。本次未改动（超出范围），建议后续改为管理员指定新密码。
