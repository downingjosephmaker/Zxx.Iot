# 插件内嵌模拟能力设计方案

> 日期：2026-07-12
> 状态：已与用户对齐，待评审
> 目标：撤销独立的 `IotSimulator` / `IotSimulator.Core` 项目，将模拟能力内嵌进各协议插件，通过统一 `ISimulatable` 接口触发，UI 落在设备管理页。

---

## 1. 背景与动机

### 1.1 现状

- **`IotSimulator`（宿主）**：控制台程序，`dotnet run -- --scenario xxx.json` 起一个场景，Ctrl+C 退出。纯 CLI，无 UI，无法与平台联动。
- **`IotSimulator.Core`（内核）**：真正的模拟逻辑。包含 `ModbusSlave`（FluentModbus 托管寄存器）、`Dlt645Slave` / `Cjt188Slave`（一问一答编帧从站）、`FaultInjector`（故障注入）、`ScenarioRunner`（场景编排）、`SimTcpServer/Client`（传输层）、`IValueGenerator`（数值生成器）。

### 1.2 问题

1. 模拟器与平台**割裂**：读独立场景 JSON，与平台真实设备/点表两套数据源，无法一键对拍真实采集链路。
2. 传输层**重复造轮子**：`SimTcpServer/Client` 与 `IotDriverCore` 的 `TcpServerChannel`/`TcpClientChannelPool` 功能重叠。
3. 无 UI，非工程师难以使用。

### 1.3 目标

- **撤销** `IotSimulator` + `IotSimulator.Core` 两个独立项目。
- 模拟能力**内嵌进各协议插件**，插件既能当从站被平台采集（设备模拟），也能当主站对拍验证（编解码自测）——**双向**。
- 数据源**复用平台设备/点表**（`DeviceInfo` + `DeviceTypeParam`），零手工配点。
- 触发接口统一为 `ISimulatable`，经 `SimulatorController`（REST）暴露给前端。
- UI 落在**设备管理页**：设备行内加「采集」「模拟」两个按钮。

---

## 2. 架构决策（已与用户对齐）

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 模拟角色 | **双向**（从站被采集 + 主站对拍） | 最完整，覆盖设备模拟与编解码自测两类场景 |
| 数据源 | **复用平台设备/点表** | 与真实采集链路天然对齐，零手工配点 |
| 触发接口 | **新增 `ISimulatable` + REST 控制器** | 最清晰、可测、易扩展 |
| 模拟落位 | **方案 A：模拟人格独立运行**（独立端口/生命周期，不与采集通道纠缠） | 隔离干净、符合 SOLID 单一职责 |
| 编解码复用 | **保持独立实现**：从站迁入插件物理目录，但编解码仍用自己那套，**不调插件 FrameHelper** | 保留原作者的"对抗性验证"设计意图：模拟与插件互为独立实现，才能测出编解码对称性 bug |
| 采集按钮语义 | **启用/停用该设备采集**（切 `IsCollection` + 触发插件配置更新） | 语义清晰，复用现成 `ConfigReloadNotifier` |
| 模拟按钮体验 | **弹窗**：选参数后一键起模拟，含实时帧日志 | 可见故障注入与帧日志，体验完整 |
| 全局视野 | **模拟弹窗内加"全部运行中"小标签页** | 兼顾单点操作与全局管理，符合 YAGNI |

## 3. 后端设计

### 3.1 新接口 `ISimulatable`（`IotDriverCore/Contracts/ISimulatable.cs`）

模拟能力是**可选人格**，插件按需实现，不强制所有插件都实现。

```csharp
namespace IotDriverCore
{
    /// <summary>
    /// 协议模拟人格契约(插件可选实现;与采集能力共存于同一插件实例,
    /// 但独立端口/独立生命周期,不与采集通道纠缠——方案A隔离原则)
    /// </summary>
    public interface ISimulatable
    {
        /// <summary>模拟能力元信息(支持的模式/默认端口等,前端渲染用)</summary>
        SimCapability Capability { get; }

        /// <summary>启动模拟(传入选中的设备+点表快照+运行参数,返回运行句柄状态)</summary>
        Task<SimStatus> StartSimAsync(SimStartRequest request, CancellationToken ct);

        /// <summary>停止指定模拟实例</summary>
        Task StopSimAsync(string simId);

        /// <summary>查询当前所有模拟实例状态</summary>
        IReadOnlyList<SimStatus> ListSims();

        /// <summary>注入/清除故障(错帧/超时/半包等,复用现有 FaultInjector 语义)</summary>
        Task InjectFaultAsync(string simId, SimFaultSpec fault);

        /// <summary>实时日志回调(收发帧 hex 摘要,宿主接到后转推 SignalR)</summary>
        Action<SimLogEntry>? OnSimLog { get; set; }
    }
}
```

配套数据契约（`IotDriverCore/Contracts/SimModels.cs`）：

- `SimCapability`：`{ SupportSlave, SupportSelfTest, DefaultPort, Protocol }`
- `SimStartRequest`：`{ Mode(Slave/SelfTest), Port, Devices[]（含点表快照）, GeneratorOverrides[], Faults[] }`
- `SimStatus`：`{ SimId, DeviceId, Mode, Running, Port, StartedAt, Message }`
- `SimFaultSpec`：`{ Kind(错帧/超时/半包), Param }`
- `SimLogEntry`：`{ SimId, Time, Direction(←/→), Hex, Note }`

### 3.2 双向能力实现

- **从站方向**（被平台采集，主战场）：`Mode = Slave` 时起监听从站，平台采集插件连上来读到假数据。
- **主站方向**（编解码自测）：`Mode = SelfTest` 时插件用自己的 `ReadBatchAsync`/编解码对内置假从站跑读写，验证编解码正确性，结果走 `SimStatus.Message` 回报。

### 3.3 从站逻辑迁移（三类资产分流）

| 资产 | 去向 | 理由 |
|------|------|------|
| `ModbusSlave` / `Dlt645Slave` / `Cjt188Slave` | **迁入各自插件** `IotPlugin.Xxx/Sim/XxxSlave.cs`，编解码逻辑整体搬家，**不改为调用插件 FrameHelper** | 保留对抗性验证：模拟从站与插件采集器互为独立实现 |
| `IValueGenerator` / `GeneratorFactory` / `FaultInjector` | **下沉 `IotDriverCore/Simulation/`** | 协议无关公共设施，5 插件共享，DRY |
| `GeneratorModel` / `FaultModel`（原在 `ScenarioModel.cs`） | **随生成器/故障注入下沉 `IotDriverCore/Simulation/SimGeneratorModels.cs`** | 生成器/故障注入依赖它们；`ScenarioModel` 其余部分删除，故须裁剪出这两个轻量模型独立下沉 |
| `SimTcpServer` / `SimTcpClient` | **删除，复用 `TcpServerChannel`** | 重复造轮子，采集已在用这套通道 |
| `ScenarioModel` / `ScenarioLoader` / `ScenarioRunner` | **删除** | 数据源改为平台设备/点表 |

**依赖成本（诚实提醒）**：`ModbusSlave` 依赖 `FluentModbus` NuGet 包，迁入后 `IotPlugin.Modbus.csproj` 需新增该包引用。其余插件按其从站实现的实际依赖补充。

### 3.4 数据源接入（复用平台点表）

```
用户在 UI 勾选设备
  → SimulatorController 从 DB 查 DeviceInfo + DeviceTypeParam(点表)
  → 组装 SimStartRequest.Devices[]（含每点 FuncCode/地址/数据类型）
  → 插件 StartSimAsync 为每点创建 IValueGenerator
  → 从站监听端口，按点表应答假数据
```

**字段映射**：现有 `IotSimulator.Core` 的 `DeviceModel/PointModel`（场景 JSON）与平台 `DeviceInfo/DeviceTypeParam` 字段名不同。迁移时写轻量映射（`DeviceTypeParam.CollectFuncCode → PointModel.FuncCode` 等），**从站内部逻辑不改**，只换数据来源，迁移风险最小。

### 3.5 值生成策略（渐进式）

- **默认**：按数据类型给合理随机游走值（温度类 20~30 浮动、开关类 0/1 翻转），开箱即用。
- **可选覆盖**：UI 允许对单点指定生成器（正弦/斜坡/固定值/随机），存前端临时参数，**不落库、不污染点表**。

### 3.6 日志实时回流

复用现成 SignalR（`ChatServer`）。模拟从站收发帧 hex、故障注入事件经 `OnSimLog` 回调 → 宿主 → SignalR 推前端模拟控制台。**不新造通道**（YAGNI）。

### 3.7 REST 控制器（`Areas/Device/Controllers/SimulatorController.cs`）

统一路由到插件 `ISimulatable`，从 `OperatorCommon.DicPlugins` 取实例转型调用。

| 端点 | 作用 |
|------|------|
| `POST Api/Simulator/GetDeviceSimMeta` | 传 deviceId，返回点表快照 + 所属插件是否支持模拟 + 默认端口 |
| `POST Api/Simulator/StartSim` | 传 deviceId + 端口 + 生成器覆盖 + 故障项，路由到 `StartSimAsync` |
| `POST Api/Simulator/StopSim` | 传 simId，停止 |
| `POST Api/Simulator/ListSims` | 查所有运行中模拟实例 |
| `POST Api/Simulator/InjectFault` | 运行中注入/清除故障 |

### 3.8 采集开关（`DeviceInfoController` 轻量新增）

```
POST Api/DeviceInfo/ToggleCollection  { deviceId, isCollection }
→ 更新 IsCollection 单字段 + 触发所属插件配置更新(ConfigReloadNotifier 现成)
```

`ConfigReloadNotifier` 已在 `DeviceInfoController` 构造函数注入，改完 `IsCollection` 顺手通知插件重建拓扑，采集立即生效。

## 6. 非目标（YAGNI）

- 不做独立模拟中心页（`/iot/simulator`）——用户明确要落在设备页。
- 不做生成器参数落库——临时覆盖足够，避免污染点表。
- OpcUa/S7 从站模拟暂不强制实现——`ISimulatable` 可选，后补。
- 不保留场景 JSON 导入/导出——数据源已改为平台点表。
