# 协议通用工具库与脚本注入方案

> 目标：让 JS 协议脚本作者不必手搓 CRC 表，同时收敛 C# 插件里的重复实现。
> 关联文档：[IEC104协议插件方案.md](./IEC104协议插件方案.md)（依赖本方案的 `decodeValue`）

## 一、背景与根因

### 1.1 脚本侧：能力存在，但脚本够不着

`ScriptSandbox`（`IotDriverCore/Scripting/ScriptSandbox.cs`）刻意不开 `AllowClr`，沙箱内是纯 JS。
脚本作者要做 CRC16 校验，只能在 JS 里手抄一张 512 字节查表——这就是"脚本解析太难"的直接来源。

而 `CenBoCommon.Zxx` 里 `YjXieYiCheck`（CRC16 四变体 + 累加和）、`YjJinZhiChange`（进制转换 + Hex↔byte[] + BCD）**早已存在，且被全部插件的 csproj 引用**。能力是有的，只是没接到脚本上。

### 1.2 C# 侧：公共库缺了一块 API，所以插件只能自己写

`YjXieYiCheck.YjSumByte(byte[] data)` 只能算**整个数组**的和，没有 `(offset, count)` 重载。
而 645/188 的帧校验必须算「从第一个 `0x68` 起、到 `CS` 之前」的一段（前面还有 `FE` 唤醒前导要跳过）。

**插件作者不是偷懒，是公共库的 API 缺口。** 这个缺口不补，收敛就是硬掰。

### 1.3 一处从未通电的 bug

`CenBoCommon.Zxx/YjJinZhiChange.cs:214`，`ConvertGenericBinaryFromDecimal` 的 `case 16:` 分支是空的：

```csharp
case 16:
    break;          // ← 什么都没做，output 保持 ""
```

即 `ConvertJzChange(input, 10, 16)` **从来没有转换成功过**，一直返回空串。

## 二、分层归属

| 层 | 位置 | 装什么 | 判据 |
|---|---|---|---|
| 通用层 | `CenBoCommon.Zxx`（已有） | 进制转换、CRC16、累加和 | 换个行业也能用 |
| 协议层 | `IotDriverCore/Codec/`（**新建**） | BCD 地址/值、`FE` 前导、字节序重排、类型解码 | 只有工业协议才有的概念 |
| 脚本层 | `ScriptSandbox` 注入（**新建**） | 把上两层暴露为 JS 的 `Zxx` 对象 | — |

### 为什么协议层放 `IotDriverCore` 而不是 `CenBoCommon.Zxx`

1. BCD 表地址、`FE` 唤醒前导、`+33H` 偏移是**工业协议专属概念**，塞进通用库会污染它；`CenBoCommon.Zxx` 是跨项目共享的。
2. `ScriptSandbox` 本来就住在 `IotDriverCore`。协议层放这里，注入时天然够得着，**不需要新增任何项目引用**。
3. 所有插件都已引用 `IotDriverCore`，收敛无需改 csproj。

## 三、改动清单

### 3.1 通用层 `CenBoCommon.Zxx`

| 文件 | 改动 | 性质 |
|---|---|---|
| `YjJinZhiChange.cs:214` | 补上 `case 16:` 的 10→16 实现 | **bug 修复** |
| `YjXieYiCheck.cs` | 新增 `YjSumByte(byte[] data, int offset, int count)` 重载 | **补 API 缺口**（3.2 收敛的前提） |
| `YjJinZhiChange.cs` | 新增 `ToBcdByte(int value)`（int→BCD 单字节） | 补方向缺口（现有 `ConvertBCDToInt` 只有 BCD→int） |

现有方法签名与行为**一律不改**，只做增量。

### 3.2 协议层 `IotDriverCore/Codec/` — 收敛真重复

| 提取的方法 | 现在在哪 | 相同度 |
|---|---|---|
| `DecodeBcdValue(byte[], bool signed)` | `Dlt645FrameHelper.cs:124`、`Cjt188FrameHelper.cs:120` | **逐字符完全相同** |
| `WithWakePrefix(byte[])` | `Dlt645FrameHelper.cs:170`、`Cjt188FrameHelper.cs:164` | **完全相同**（4×`FE`） |
| `CheckSum(byte[], int offset, int count)` | `Dlt645FrameHelper.cs:160`、`Cjt188FrameHelper.cs:154` | **完全相同**（模 256 和） |
| `BuildBcdAddress(long adr, int bytecount)` | `Dlt645FrameHelper.cs:30`(6字节/12位)、`Cjt188FrameHelper.cs:31`(7字节/14位) | 同构，仅字节数不同 → 参数化 |
| `DecodeBinValue(byte[])` | `Cjt188FrameHelper.cs:141` | 单处，提升供复用 |

新建文件：
- `IotDriverCore/Codec/ProtocolCodec.cs` — 上述五个方法
- `IotDriverCore/Codec/ValueCodec.cs` — `ModbusValueCodec` 提升而来（见 3.3）

原插件的 `Dlt645FrameHelper` / `Cjt188FrameHelper` 保留，内部改为委托到 `ProtocolCodec`，**对外签名不变**（`internal static`，仅插件内使用，不影响任何调用方）。

### 3.3 `ModbusValueCodec` 提升为 `ValueCodec`

`IotPlugin.Modbus/ModbusValueCodec.cs` 目前是 `internal static`，关在 Modbus 插件里。它提供：

- 字节序四选一：`ABCD` / `CDAB`（字交换）/ `BADC`（字内换）/ `DCBA`（全反转），三种重排均为对合变换
- 九种数据类型：`bool` `bit` `int16` `int32` `uint32` `int64` `float32` `float64` `bcd` `string`
- 位偏移取布尔

**这是脚本作者最需要的东西**——"4 个字节怎么变 float"的答案就在这里。IEC104 的 TI=13 短浮点数也要用它。

处置：整体移到 `IotDriverCore/Codec/ValueCodec.cs`，改为 `public static`，**并删除 `IotPlugin.Modbus/ModbusValueCodec.cs`**，改掉 `ModbusPlugin` 内的调用点与 `ModbusValueCodecTests` 的引用。

**不保留薄转发。** 提升后的 `ModbusValueCodec` 若只剩转发，就是一个新造的空壳——本仓已有太多"结构完整、从未通电"的角落，不该再添一个。

> 与 3.2 的区别：`Dlt645FrameHelper` / `Cjt188FrameHelper` **保留**，因为它们除了被提取的那几个方法外，还有 `BuildReadFrame` / `TryParseFrame` 等协议专属逻辑，委托后仍是有实质内容的类。而 `ModbusValueCodec` 被整体提升后一无所剩。

### 3.4 Modbus 的 CRC：**不动**

`ModbusFrameHelper.cs:1` 首行即 `using CenBoCommon.Zxx;`，`AppendCrc` 内部调 `pdu.YjCrc16LHBtye()`，`TryParseRtu` 调 `body.YjCrc16VerifyLH(crc)`。
**Modbus 的 CRC 早已收敛到公共库**，那个 7 行 `AppendCrc` 只是薄封装，不是重复实现。

### 3.5 绝对不碰的（重要）

```
IotPlugin.Dlt645/Sim/Dlt645Slave.cs  → CheckSum / EncodeBcdValue / BuildAddressBcd
IotPlugin.Cjt188/Sim/Cjt188Slave.cs  → CheckSum / EncodeBcdValue / BuildAddressBcd
```

看着是重复，**其实是故意的**：模拟器从站独立实现编解码，是为了与插件主体互相做**对抗性验证**。
两边共用同一份代码，`SlaveRoundTripTests` 就退化成"自己证明自己"，验证价值归零。

**这几处一行都不许动。**

## 四、脚本注入设计

### 4.1 一条铁律

**所有注入方法只收发 `string` / `number` / `bool`，绝不碰数组、绝不返回 CLR 对象。**

- 前半条：`ScriptSandbox.cs:26` 的 Prelude 注释白纸黑字写着"规避CLR数组封送歧义"——原作者已踩过这个坑。
- 后半条：一旦某方法返回 CLR 对象，Jint 会将其包为 `ObjectWrapper`，暴露面悄然扩大。

### 4.2 API 清单

| 分组 | 方法 | 实现归属 |
|---|---|---|
| 字节↔hex | `toHex(u8)` / `fromHex(hex)` | **纯 JS Prelude**（参数是 `Uint8Array`，封送雷区） |
| 校验 | `crc16(hex)` `crc16HL(hex)` `verifyCrc16(hex, crc)` `sum(hex)` `verifySum(hex, cs)` | C# → `YjXieYiCheck` |
| 进制 | `hex2dec` `dec2hex(n, len)` `hex2bin` `bin2hex` `bin2dec` `dec2bin` | C# → `YjJinZhiChange`（含修好的 10→16） |
| BCD | `bcd2dec(hex)` / `dec2bcd(n, bytes)` | C# → `ProtocolCodec` |
| 取值 | `decodeValue(hex, type, order, bit)` / `encodeValue(v, type, order, len)` | C# → `ValueCodec` |

`crc16` 默认低字节在前（Modbus 线序），`crc16HL` 为高字节在前。

### 4.3 脚本样例

```js
function decode(frame, ctx) {
  var hex = Zxx.toHex(frame);

  if (!Zxx.verifySum(hex.substr(2, 20), hex.substr(22, 2))) return null;

  var kwh = Zxx.decodeValue(hex.substr(24, 8), "float32", "DCBA", -1);
  return { telemetry: [{ key: "energy", value: parseFloat(kwh) }] };
}
```

### 4.4 实现方式

与现有 `console` 注入完全同构（`ScriptSandbox.cs:59`）：

```csharp
_engine.SetValue("Zxx", new ScriptCodecApi());
```

`ScriptCodecApi` 是一个 POCO，公共方法即 JS 可见方法。**不开 `AllowClr`。**

### 4.5 安全边界论证

Jint 4.11.0 官方源码（`Jint/Options.cs`、`Jint/Runtime/Interop/TypeResolver.cs`）：

> `Options.Interop.AllowGetType` **默认 `false`**。官方注释："Whether to expose `object.GetType()` which can allow bypassing allow lists and open a way to reflection."
> `TypeResolver.Filter` 据此拦截 `GetType` 成员访问。

结论：
- `System` 仍为 `undefined` → `ScriptSandboxTests.CLR互操作被禁用_无法访问宿主类型` 继续绿。
- 脚本无法顺着 `Zxx` 反射回 CLR。
- 暴露面与现有 `console` 注入**同级**，不引入新的风险类别。

**加固措施**：在 `ScriptSandbox` 构造的 options 中**显式**写出 `options.Interop.AllowGetType = false`。虽与默认值相同，但把安全依赖从"库的默认行为"变成"本代码的显式声明"，防止将来 Jint 升级改默认值时静默失守。

### 4.6 性能

CRC 查表在 C# 侧执行，**不计入脚本 `MaxStatements` 的 10 万条限额**（`ScriptSandbox.cs:23`）。
若改用纯 JS 实现，每帧 CRC 会吃掉数百条语句配额。

## 五、风险

### 5.1 ALC 类型分裂（本仓爆过的雷）

往 `IotDriverCore` 加新类型后，若插件目录旁置了**陈旧的** `IotDriverCore.dll`，会触发跨 `AssemblyLoadContext` 类型分裂——典型症状是莫名报"某接口未实现"，排查极其痛苦。

缓解：`PluginLoadContext.StripSharedAssemblies` 已做上传时剥离加固。
**但仍要求：本方案落地后，所有在线插件包必须全部重新编译上传，不得只传改动的那个。**

### 5.2 收敛引入行为漂移

`DecodeBcdValue` / `CheckSum` / `WithWakePrefix` 虽逐字符相同，提取过程仍可能手滑。

缓解：`Dlt645FrameHelperTests`、`Cjt188FrameHelperTests`、`SlaveRoundTripTests`、`ModbusValueCodecTests`、`ModbusFrameHelperTests` 已存在且覆盖这些路径。**收敛前后测试数与通过数必须完全一致**，任何一条变红即回退该项。

## 六、验证

1. `dotnet build` — 0 错误
2. `dotnet test` — 全绿，且**用例总数不减少**
3. 新增用例：
   - `ConvertJzChange(x, 10, 16)` 返回正确十六进制（**修复前必然失败**，先写红再修绿）
   - `YjSumByte(data, offset, count)` 与 `Dlt645FrameHelper` 原 `CheckSum` 逐字节等价
   - `ScriptSandbox` 注入后 `Zxx.crc16("0103000A0002")` 与 C# 侧 `YjCrc16LHStr` 结果一致
   - `ScriptSandbox` 注入后 `System` 仍为 `undefined`（安全边界回归）
   - 脚本内 `Zxx.GetType` 不可用（反射逃逸回归）

## 七、不做什么（YAGNI）

- **不动 `Sim/*Slave.cs`**（见 3.5）。
- **不动 Modbus CRC**（见 3.4）。
- **不重写 `CenBoCommon.Zxx` 现有方法签名**，只做增量。那些拼音命名（`YjCrc16HLStr` 等）虽不理想，但改名会波及全仓调用点，与本次目标无关。
- **不给 `encode` / `splitFrames` 补生产链路**。经查证，二者当前仅 `ProtocolScriptService.DryRun` 调用，生产路径只有 `decode`（`DecodeMqttPayload`）在跑。补全它们是独立议题，不在本方案范围。
- **不做前端「内置函数速查」面板**。有价值，但属范围扩张；待 `Zxx` 落地、脚本作者反馈后再议。
