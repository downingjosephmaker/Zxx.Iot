## 0. 一句话结论(先给判断,再给依据)

P3 的真实风险画像**比总体方案文档乐观**。四路探子用代码实证纠正了两条过度保守的假设,并揪出一条真正的杀手:

| 文档原假设 | 探子实证结论 | 对 P3 的影响 |
|---|---|---|
| 改列名 → 旧 Token 失效 → **强制全员重登** | **虚惊**。Token 存的是 `{"UnitId":5}`(属性名+数值),列名从不进 Token。改列名对 Token 零影响 | **"强制重登"约束可删**(除非另改属性名) |
| CodeFirst 不自动改列,需手动 DDL | 属实,且更危险:CodeFirst 会**抢跑加空 `tenant_id` 列 + 留孤儿 `unit_id` 列**,直接丢数据 | **真正的杀手**,必须"停服+手工 RENAME 先行" |
| 改列名波及隔离逻辑 | **不波及**。隔离全链路只认属性名 `TenantId`+接口,零硬编码列字符串 | 风险面仅 17 个特性 + 分表 + hypertable + DDL 本身 |
| 租户树要从零设计 | P0 删的 BuildInfo/DeptInfo(`2dba029^`)是**现成模板** | 树形逻辑可直接照搬 |

## 1. 侦察情报汇总(四本账)

### 1.1 物理列 unit_id 散落点(探子①)

`unit_id` 字面量全仓仅出现在:**17 个 `[SugarColumn]` 特性** + **1 个手写 DDL 文件** + 1 个设计文档。**C# 内零处拼接/Ado 原生 SQL 含 `unit_id`**——无隐藏漏网。

**受影响物理表 4 类 26 张:**

**A 类 · 隔离歧视列(22 张,`RENAME COLUMN unit_id→tenant_id`):**
直接映射 12 张:`device_info` / `device_param` / `device_comfort` / `device_alarm_config` / `device_type_run` / `device_type_alarm_config` / `scada_project` / `dash_project` / `dash_data_pool` / `admin_mqttparam` / `admin_logoparam` / `sys_related`。
EventBase 派生 9 事件表:`event_alarm` / `event_control` / `event_history` / `event_run` / `event_signal` / `event_peak_day` / `event_report_day` / `event_report_week` / `event_report_month`。
> ⚠️ `event_history` / `event_run` / `event_signal` 走 **SqlSugar SplitTable 分表**,物理上有多张按日/月分表(如 `event_history_20260710`),RENAME 须**对每个已存在分表实例逐个执行**。

**B 类 · 属性名仍是 UnitId(3 张,改需连带 C# 引用):**
`sys_user`(牵登录链 LoginPublicMode/OperatorCommon)/ `push_strategy` / `collect_strategy`(后两者注释"平台级共享,暂不挂 ITenantEntity")。

**C 类 · 主键列(1 张,建议保留不改):**
`basicunit_info` 的 `unit_id` 是 `IsPrimaryKey+IsIdentity` 自增主键,被所有表外键指向。改它牵连自增序列+全部外键,**建议保留 `unit_id` 作 PK 不动**,只改 A/B 类歧视列。

**D 类 · 手写 DDL / hypertable(1 张,SqlSugar 之外):**
`iot_ts.device_event_log`(`database/timescaledb/V0005__event_logs.sql:17`)。**不走 CodeFirst**,须在 V0005 脚本+实际库手动 RENAME,且是 hypertable,须确认 TimescaleDB RENAME COLUMN 兼容性。

### 1.2 CodeFirst 建表机制(探子②)

- `InitTables()` 主方法在 `Program.cs:19` **被注释掉**,是死代码。真实建表靠**每实体首访时懒触发 CodeFirst**(`DbContext.cs:868`),开关 `DbSetting.Migration` **默认 = true**(`DbSetting.cs:14`)。
- CodeFirst 是**增量**的:遇到"属性映射列名从 unit_id 变 tenant_id",默认行为 = **ADD 空 `tenant_id` 列(默认0)+ 不删旧 `unit_id` 列(PG 路径不 drop)**→ 新列全空、老数据成孤儿、业务读 `tenant_id` 全 0 = **数据丢失**。
- 隔离过滤器物理触点**仅 2 处**:(a) 各实体 `SugarColumn.ColumnName` 字符串 (b) 库里真实列。过滤/回填/缓存链走属性名,**自动跟着 ColumnName 走,不用动**。

### 1.3 Token 链路(探子③)

- Token = `OperatorModelLogin` 对象的 DES 加密 JSON,字段 `"UnitId":数值`(`LoginPublicMode.cs:68` 整体加密,`ToJson` 无改名特性)。
- 解密注入:`TokenAuthorizationFilter.cs:37,55` 反序列化读 `model.UnitId` → 注入 `CurrentTenantId`。
- **改物理列名与 Token 二进制内容零关系**——旧 Token 照样解密、拿到值、注入。
- 唯一会让旧 Token 失效的是把 `OperatorModelLogin.UnitId` **属性名**改成 `TenantId`(JSON key 变)——**P3 不做这一步**。即便要做,加 `[JsonProperty("UnitId")]` 兼容特性即可零成本兜底,无需重登。

### 1.4 租户树(探子④)

- `basicunit_info` 现为扁平结构,主键 `unit_id` 自增,**无任何父子字段**,带 `[EntityCache]` 全表 Redis 缓存。需新增 `parent_id/tree_level/full_code/full_name/has_child`。
- 过滤器升级点**仅 2 行**:`DbContext.cs:781`(SQL 路径 `it.TenantId == X`)+ `:803`(缓存路径),都改成 `子孙集.Contains(TenantId)`。
- 子孙集策略:项目**已有** `OperatorModel._UnitIdList` + `BasicunitinfoController.GetQxListByPage` 的 `IN` 模式先例 → 应在**登录/上下文装配时预算"当前租户+所有子孙"的 `List<int>`** 塞进 TenantScope(把 `CurrentTenantId` 从 `int?` 扩成同时暴露子孙集),**不在过滤器里递归查库**(避免放大 DB 压力;单位表带缓存,子树可内存算)。子孙集**不进 Token**(会过期),每请求按 `parent_id` 现算。
- CRUD 主战场:`BasicunitinfoController.SaveBatch`(:39 以 UnitId==0 分插入/更新→需加"选父级+维护 tree_level")+ `DeleteByPk`(:134→需级联删子孙)。
- **现成模板**:`git show 2dba029^:IotModel/MEntity/Basic/Buildinfo.cs` 带完整 `tree_level/parent_id/full_name/full_code/has_child`;`BuildinfoController.cs:69-81` 用 `full_code.Contains("|{id}|")` 祖先链一次删整棵子树。可直接照搬。

## 2. P3 执行剧本(分步骤,每步可回滚)

> 铁律(探子②实证):**DDL 必须先于代码,代码必须先于 CodeFirst 触发**。反过来"改代码靠 CodeFirst"= 加空列+留孤儿+丢数据,**禁止**。

### 阶段 P3-A · 物理列迁移(unit_id → tenant_id)

**前置门槛(硬性):**
1. **备份 PG 库** `6305/zhjngkdb`(总体方案 §6 硬门槛)。备份到 `database/backup/`。
2. **现场核对**:用 `information_schema.columns` 确认 26 张表里**实际含 `unit_id` 列**的表清单(尤其 push/collect_strategy 注释说"预留",需实证有无);列出 `event_history/run/signal` 的**所有已存在分表实例**。
3. **确认 `Config/DbSetting.config` 里 `Migration` 运行时取值**(默认 true)。若为 true,迁移窗口内必须先停服,防 CodeFirst 抢跑。

**执行顺序:**
1. **停服**(Iot 后端 13699 下线)。
2. **跑手工 DDL**(已交付 `database/business/V0001__rename_unit_to_tenant.sql` + 对称回滚 `V0001__rollback_tenant_to_unit.sql`,PG `DO $$` 块,幂等):
   - A 类固定表 21 张 + B 类 3 张:`DO` 块内 `information_schema` 判定 + `ALTER TABLE %I RENAME COLUMN unit_id TO tenant_id`。含 CodeFirst 误加兜底(tenant_id 与 unit_id 并存时先 DROP 空 tenant_id 再改名)。
   - 分表实例:PL/pgSQL 动态枚举 `event_history_YYYYMMDD`(按日)/`event_run_YYYYMM`/`event_signal_YYYYMM`(按月),正则匹配含 unit_id 的实例逐个改名(同库)。
   - C 类 `basicunit_info`:**不改**(PK 保留,刻意排除在表数组外)。
   - **D 类 `iot_ts.device_event_log`(hypertable)暂缓**:该表走 V0005 手写 DDL、依赖 `TimescaleConString`(bin 配置为空 → 遥测服务可能未启用、表可能不存在);且 TimescaleDB 对已压缩 hypertable RENAME COLUMN 兼容性需现场测。**留 P3-6 停服时现场核实表是否存在再决定**:不存在则跳过;存在则单独出 `V0003` + 同步改 V0005 源脚本。
   - RENAME 是元数据操作,**数据原地保留,零搬运零丢失**。
3. **改代码**:把 A/B 类实体的 `[SugarColumn(ColumnName="unit_id")]` → `"tenant_id"`(17 特性中的 A/B 类;C 类 basicunit_info 主键不动)。B 类 sys_user 连带核对 `LoginPublicMode/OperatorCommon` 引用(属性名 UnitId 可不改,只要 ColumnName 对上库)。
4. **启动**:CodeFirst 发现"模型列=库列",无差异不动表,安全。
5. **验证**:登录 + SQL 日志确认过滤器注入 `WHERE "tenant_id"=X`(此前 P1+P2 验证是 `unit_id=1`,改后应变 `tenant_id`)。

**回滚**:代码 `git revert`;库用 `RENAME COLUMN tenant_id TO unit_id` 反向(或恢复备份)。

**决策点 A1**:B 类 3 张(sys_user/push/collect_strategy)是否本批一起改?
- 一起改:语义统一,但 sys_user 牵登录链,风险略高。
- 暂缓:先只改 A 类 22 张(纯隔离歧视列,零 C# 连带),B 类下批。**推荐暂缓 B 类**,把 P3-A 缩到最小安全集(A 类 + D 类)。

### 阶段 P3-B · 租户树引入

> 可与 P3-A 同一停服窗口,也可拆成独立批(推荐独立,降单批复杂度)。

1. **实体加字段**(照搬 `2dba029^:Buildinfo.cs` 模板):`Basicunitinfo` 加 `parent_id`(int,根=0)、`tree_level`(int)、`full_code`(varchar200,祖先链 `|1|3|7|`)、`full_name`(varchar400)、`has_child`(bit)。
2. **DDL**:`database/business/V0002__tenant_tree.sql` 给 `basicunit_info` `ADD COLUMN`(幂等,`AddColumnIfNotExists` 语义)。已有数据 `parent_id=0/tree_level=1/full_code=|{unit_id}|`。
3. **过滤器升级 2 行**:`DbContext.cs:781` + `:803` 从单值等于 → `子孙集.Contains(TenantId)`。
4. **子孙集装配**:`TenantScope` 从 `int?` 扩成同时暴露子孙集 `List<int>`;在 `TokenAuthorizationFilter`/`OperatorCommon` 装配上下文时,按 `parent_id`/`full_code` 从缓存的单位全表算出"当前租户+所有子孙",塞进上下文。
5. **CRUD 改造**:`BasicunitinfoController.SaveBatch` 加"选父级+算 tree_level/full_code+维护父级 has_child";`DeleteByPk` 用 `full_code.Contains("|{id}|")` 级联删子孙(照搬旧逻辑)。
6. **验证**:建父子租户,登录子租户只见自己;登录父租户见父+子(IN 集合生效)。

**回滚**:`git revert` + `DROP COLUMN` 反向(树字段是纯新增,回滚安全)。

**决策点 B1**:父子权限继承的方向?
- 方案①(推荐,旧逻辑一致):父租户可见"自己+所有子孙"数据(`IN 子孙集`),子租户只见自己。
- 方案②:仅同级隔离,父不下钻。
需昴君定,决定子孙集算法(向下收集 vs 精确匹配)。

## 3. 风险与门槛清单

| 风险 | 等级 | 缓解 |
|---|---|---|
| CodeFirst 抢跑加空列丢数据 | 🔴 高 | DDL 先于代码先于启动;停服窗口;Migration 开关核实 |
| 分表实例遗漏 RENAME | 🟠 中 | 迁移前枚举所有 `event_*_日期` 分表 |
| hypertable RENAME 不兼容 | 🟠 中 | 现场测 TimescaleDB `ALTER...RENAME COLUMN` |
| basicunit_info 主键误改 | 🟠 中 | 明确 C 类保留不改 |
| 旧 Token 失效 | 🟢 低(虚惊) | 不改属性名即无影响;要改则加 `[JsonProperty]` |
| 子孙集放大 DB 压力 | 🟢 低 | 登录时预算,走缓存,不在过滤器递归查库 |

**硬门槛**:① 备份 PG 库 ② 停服窗口 ③ 现场核实分表清单+Migration 开关+hypertable 兼容,缺一不可动手。

## 4. 决策定稿(昴君裁定 2026-07-10)

- **决策 A1 · 迁移范围**:✅ **全量 A+B+D(26 张)**。含 B 类 sys_user/push_strategy/collect_strategy。
  > ⚠️ 新增风险:sys_user 牵登录链(LoginPublicMode/OperatorCommon),须连带核对 C# 引用。对策:每改一层跑编译+登录验证。
- **决策 B1 · 权限继承方向**:✅ **父见子孙 `IN(子孙集)`**。父租户可见"自己+所有子孙",子租户只见自己。照搬旧 BuildInfo 的 `full_code` 祖先链算法(向下收集子孙)。
- **决策 C1 · 批次**:✅ **同窗口一次做**(P3-A 物理迁移 + P3-B 租户树)。单批改动大,以充分验证关卡弥补。
- **决策 D1 · 迁移方式**:✅ **版本化 SQL 脚本 + 停服人工执行**(`database/business/V0001__rename_unit_to_tenant.sql`)。
  > **纠偏记录(2026-07-10)**:初裁"DAO 幂等自愈 DDL",蕾姆亲读 `DbContext.cs:866-874` 构造函数后发现两处硬伤,昴君采纳建议改回 SQL 脚本:
  > 1. **CodeFirst 恒在 Init() 之前**:第 868 行 `Db.CodeFirst.InitTables<T>()` 先跑,第 869 行 `Init()`(自愈所在)后跑。自愈 RENAME 永远抢不到先手,只能靠"DROP 空 tenant_id 再 RENAME"打补丁,不够干净。
  > 2. **分表实体天然失效**:`event_history/run/signal` 走 `IsSplitTable` 分支(第 873 行),**根本不调 `Init()`**。DAO 自愈对这 3+张核心上行分表完全无效。
  > **结论**:版本化 SQL 脚本在停服窗口、CodeFirst 触发前一次性 RENAME 全部表(含分表实例),数据原地保留,零 CodeFirst 干扰,分表可覆盖。可审计、可回滚,补业务表无版本化迁移的空白,与现有 `timescaledb/V000x` 风格一致。
