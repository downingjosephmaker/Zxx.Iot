# 协议模拟器与插件体系完善方案（对标 IoTGateway 企业版）

> 版本：v1.0 草案　日期：2026-07-11　状态：**待评审，未实施**
> 覆盖三项需求：① 协议解析模拟器能力；② 插件在 UI 中无法添加（前端关联性/实用性差）；③ 对标 http://iotgateway.net/docs/enterprise/intro 完善系统。
> 本方案基于 2026-07-11 对全仓代码的实证勘察（7 路并行扫描，覆盖 IotDriverCore / 5 个 IotPlugin.* / IotWebApi / vuefrontend / docs / IoTGateway 全量文档），所有现状结论均有文件级证据。

**关键修正**：一期文档 `docs/IoT平台功能方案.md` §2 的"数据上行链路断裂（最严重）/AlarmConfig 无引擎"结论**已过时**——代码实态中上行管道（`DataPointIngestService`）、告警引擎（`AlarmEngineService`）、遥测写入（`TelemetryWriteService`）、联动/北向均已实现闭环。本次真正的缺口不在数据链路，而在**可测试性（模拟器）与可配置性（插件 UI/元数据）**。详见 §1.2 勘误。

**即刻可用**：不必等第二批交付——手头已开发的插件今天就能装进系统（Swagger 上传 + 改状态 + 重启后端），分步指引见 §5.6。

## 1. 现状核查

### 1.1 插件体系实态（与一期方案设想的偏差）

```
                    ┌────────────────────────── IotWebApi（宿主）──────────────────────────┐
 sys_plugin 表 ──→ PluginService ──ALC加载──→ ICenBoPlugin 实例（DicPlugins 静态字典）        │
                    │                              │                                        │
                    │              ┌───────────────┴───────────────┐                        │
                    │        IotPlugin.Modbus/Dlt645/Cjt188        IotPlugin.S7/OpcUa       │
                    │        （走 IotDriverCore 通道+指令引擎）      （库直连，自带IO循环）     │
                    │               │ PluginEvent（上行）                                    │
                    │               ▼                                                       │
                    │        DataPointIngestService ─→ 过滤/最新值/历史/遥测/告警/联动/北向/SignalR
                    └────────────────────────────────────────────────────────────────────────┘
```

- 插件真实契约是 `CenboEventBus/CenBoPlugin/ICenBoPlugin.cs`（6 个自描述属性 + Init/Start/Stop + 双向消息）。
- `IotDriverCore/Contracts/` 三件套（`IDeviceDriver`、`DriverInfoAttribute`、`ConfigParameterAttribute`）是**死契约**：全仓零实现、零反射消费。注释宣称的"平台反射生成前端配置表单"没有任何对应实现。
- 插件加载链：`sys_plugin` 表（PluginStatus==1）→ `PluginLoadContext`（可卸载 ALC）→ 反射找 `ICenBoPlugin` → `PluginInit`（注入事件总线）→ `PluginStart`（传入 DB 的 plugin_config JSON）。

## 2. IoTGateway 参考要点（驱动与 UI 的可借鉴设计）

来自 iotgateway.net 全量文档抓取（社区版 v3.0.1 + 企业版 6.1~6.8）：

1. **驱动反射插件模型**：驱动实现 `IDriver`，用 `[DriverInfo]`（类级元数据）、`[DriverSupported("设备类型")]`（类级归属声明）、`[ConfigParameter("中文名")]`（属性级，web 端反射生成配置表单并赋值）、`[Method("方法名")]`（读取方法清单，前端"方法"下拉的数据源）。→ 我们的 `DriverAttributes.cs` 就是照此设计的，只是从未接线。
2. **设备维护 UI 流程**：驱动管理（注册 DLL）→ 设备维护（建设备时**选驱动**）→ 通讯配置（反射 ConfigParameter 出表单）→ 设备变量（变量名/方法/地址/数据类型/表达式 raw/上传策略，支持 Excel 导入）→ 启动采集 → WebSocket 实时看值。**关键差异：设备与驱动显式绑定，表单按驱动裁剪**。
3. **企业版核心增值**（对标目标）：vue 前后端分离 + **设备树单页面完成几乎所有功能**；北向插件化多路输出（PlatformHandler 工厂模式）；**断网/断电数据缓存续传**；关系库宽表（分表 + 变量变化时表结构自动迁移）；自带时序库 + 数据分析；规则流（Node-RED 式）；web 组态/大屏/报表；主机指标监控；**读取公式/写入公式分离**。

### 3.1 组态与 IoT 管理的衔接现状（2026-07-11 勘察补充）

## 4. 方案 A：协议解析模拟器（需求①）

### 4.1 目标与形态

**目标**：不接真实设备，就能对 5 个协议插件做"编帧→通道→拆帧→解析→定标→入库→推送"的全链路验证与回归；同时补齐纯函数层单测。

**形态决策（推荐 a）**：
- (a) **独立控制台项目 `IotSimulator` + 类库 `IotSimulator.Core` + 测试项目 `IotTests`**，场景 JSON 驱动，开发者本机运行——先满足工程刚需；
- (b) 产品化虚拟设备（IotWebApi 内托管 + UI 管理 + 复活 `IsVirtual` 字段）——**后置**为 C 阶段可选项，等模拟器内核被 (a) 验证后再包装。
- 理由：用户痛点是"开发的插件没法测"；产品化包装是第二步，先做会拖慢第一批交付（YAGNI）。

### 4.2 架构

```
IotSimulator.Core（类库，net10.0）
├── Transport/
│   ├── SimTcpClient      // 拨入模式：连插件的 TcpServerChannel，先发 DTU 注册包(ASCII)，可选心跳
│   └── SimTcpServer      // 被拨模式：监听端口，接受插件 TcpClientChannelPool 拨入（Modbus TCP 场景）
├── Slaves/
│   ├── ModbusSlave       // 基于 FluentModbus（一期 §4.2 既定选型）：TCP 从站直接用库；
│   │                     // RTU over TCP（DTU 拨入）先 spike 验证 ModbusRtuServer+IModbusRtuSerialPort
│   │                     // 的 TCP 流适配（注册包握手仍需自研前置），不可行再自研请求解析+应答编码
│   ├── Dlt645Slave       // 自研：按 DL/T645-2007/1997 国标独立实现从站编解码（不 import 插件 helper，见 4.3 决策）
│   └── Cjt188Slave       // 自研：CJ/T188 从站，含表型 T/SER 序列号/阀控应答
├── Generators/           // 点值生成器：constant | random(min,max) | sine(period,amp) | step | replay(csv)
├── Faults/               // 故障注入：不回复(超时) | CRC/CS错帧 | 异常码应答 | 粘包(多帧合发) | 半包(分片+延迟) | 乱序注册包
└── Scenario/             // 场景模型 + JSON 加载器
IotSimulator（控制台宿主）
└── dotnet run --project IotSimulator -- --scenario scenarios/dlt645-dtu-3meters.json
IotTests（xunit）
├── FrameSamples/*.json   // 帧样本库：标准帧/异常帧/粘包序列 hex + 期望解析结果
└── 针对 Dlt645FrameHelper / Cjt188FrameHelper / ModbusFrameHelper / ModbusValueCodec /
    FrameAccumulator / PointBatchBuilder / ScriptSandbox 的单测（插件项目加 InternalsVisibleTo("IotTests")）
```

另加一个 **MQTT 设备模拟器**（`IotSimulator.Core/Mqtt/MqttDeviceSim`）：按 `MqttClientJob` 的三种载荷契约（PluginMessage 包装 / 裸 `List<DeviceData>` / 非 JSON hex 走产品 JS 脚本解码）向 RabbitMQ 1883 发布——把历史上"每次手工重来"的端到端测试脚本沉淀成仓库资产。

### 4.3 关键设计决策

| 决策点 | 推荐 | 理由 |
|---|---|---|
| 从站编解码是否复用插件的 FrameHelper | **不复用，独立实现**（Modbus 用 FluentModbus，645/188 按国标自研从站侧） | 模拟器的价值一半在"对抗性验证"——用同一份代码自己测自己，编解码对称性 bug（如 33H 偏移、BCD 低位在前）永远测不出来 |
| 单测是否复用插件纯函数 | **复用**（InternalsVisibleTo） | 单测的目标恰是这些纯函数本身；帧样本库同时喂单测与模拟器，二者互为印证 |
| 场景配置来源 | 手写 JSON 为主；随仓提供与 `di_template_2007.sql` 对齐的 645 场景、与默认点表对齐的 Modbus 场景 | 保证"库里配好点表 + 起模拟器"即可跑通，不做点表反向导出工具（YAGNI） |
| 是否模拟 S7/OpcUa | **不自研**。S7 用 Snap7 服务端或跳过；OpcUa 单测其 NodeId 映射逻辑即可 | 两者是库直连型插件，协议栈由第三方库承担，自研模拟收益极低；文档注明用 KEPServerEX/Prosys 等现成模拟器做集成验证 |

  "transport": { "mode": "dial-in", "host": "127.0.0.1", "port": 20645,

### 4.5 验收标准（M-A）

1. `dotnet test` 全绿：每个协议 ≥15 个用例，必含异常帧（CS/CRC 错、长度域错、DI 不符）与粘包/半包序列（直测 `FrameAccumulator` 三种提取器）。
2. 端到端（`start-dev.ps1` 环境 + `di_template_2007.sql` 点表）：起 645 场景（DTU 拨入、一 DTU 挂 3 表），`/iot/monitor` 页面持续看到 3 表数据按周期刷新，值与生成器函数吻合（定标 0.01 正确）。
3. Modbus 双模式各过一遍：TCP 拨出（SimTcpServer + FluentModbus）与 RTU over TCP 拨入（注册包模式）。
4. 鲁棒性：kill 模拟器 30 秒再拉起 → 插件按 `ReconnectBackoff` 自动恢复采集，超时点位不丢队列（验证"超时重排永不删除"语义）；注入 10% 错帧 → 数据不脏（`ValueFilterService` 前置的解析层就应丢弃）。
5. MQTT 模拟器三契约各发一轮 → 入库/SignalR 推送/告警触发可观测。

### 4.6 工作量估算

内核+三从站 6~8 天（含 FluentModbus RTU 从站 TCP 适配 spike 0.5~1 天），故障注入与场景化 2 天，IotTests+帧样本库 3~4 天，MQTT 模拟器 1 天，端到端联调 2 天。**合计约 2.5~3 周（单人）**。

## 5. 方案 B：插件管理 UI 与配置元数据链路（需求②核心）

### 5.1 为什么"补一个页面"不够

需求②的表象是"UI 里不知道怎么添加插件"，实底是三层断裂：
1. **前端层**：API/页面/路由全缺（D3）——补页面能解决"看得见"；
2. **配置层**：DB `plugin_config` 是死轨（D1）——不修这条，页面上保存的配置**根本不会生效**；
3. **元数据层**：插件无自描述（D10）、`Expand_SysPlugin` 空壳——不修这条，配置表单只能是裸 JSON 文本域，"太差劲"的体验换个地方重演。

三层必须一起修，顺序为 2 → 3 → 1。

### 5.2 后端整改（B-1，前置于前端页面）

| 任务 | 设计 | 涉及文件 |

3. **点表表单按驱动裁剪**：依据 = 方案 B-1.3 的 Manifest（插件认领 DeviceTypeCodes → 推导 产品→驱动 映射，**不新增表字段**）；有映射时只显示对应采集字段组（modbus 组 / opcua 组 / …），无映射时保持现状全显。多插件认领同一类型编码时显示告警提示（配置冲突本身就该暴露）。
4. **保存即生效配套**（收编 D9）：设备/点表保存后，经**既有** `IEventBus<PluginCommandEvent>` 通路（`PluginCommandEventHandler` → `ReceiveMessageAsync`，与设备控制同路）发 `PluginMessageEnum.配置更新`（枚举值 9 已存在，插件端为新增分支）；分支实现为**插件内部全量重启（PluginStop→PluginStart）**，与 B-1.7 同一语义。不做增量刷新：`RebuildCollectQueues`/`RefreshBindings` 只能替换既有端点的队列，覆盖不了通道拓扑变化（通道/引擎只在 PluginStart 创建、`TcpClientChannelPool` 无 RemoveEndpoint、零设备启动直接返回 false），增量调和的成本远高于全量重启，有性能诉求再演进。
5. U5：monitor 设备选择改为树选组件 + 远程搜索。
6. **C-6 组态 IoT 数据源绑定**（收编一期 M5-5.3 数据池收敛，已拍板 2026-07-11）：组态数据集面板（`DatasetPanel.vue`）新增数据集类型 **"IoT 设备点位"并设为默认**——绑定器 = 设备树选择（复用 C-1 树组件）→ 点位多选（按设备产品点表 `device_type_param`，显示参数名/编码/单位）→ 模式选择（实时/历史）。运行时（`runtime.vue`）：实时模式 = `GetDeviceLatest` 铺底 + SignalR `device:{id}` 组增量推送（与 monitor/设备中心同通道，同设备多图元共享一份订阅）；历史模式 = 遥测查询接口（TimescaleDB `telemetry`/`telemetry_1h`，接口若缺随设备中心"历史曲线"tab 一并新建）。数据经平台 API 自动携带 token 与租户过滤，关闭"任意 URL 无收敛"缺口。存量 HTTP/MQTT 数据源类型**保留兼容**（外部系统数据仍需通用源），仅不再作为设备数据的推荐路径（Q8）；匿名分享的无 token 数据通道仍属一期 M5-5.2，不在本项。

### 6.3 验收标准（M-C）

1. 从零配置一台 645 表：设备中心单页内完成 建产品→导 DI 模板→建设备→配策略→看实时数据 全流程，**中途不手输任何编码、不切换页面**；配合方案 A 模拟器出数。
2. 点表表单在 Modbus 产品下不出现 NodeId 字段，反之亦然。
3. 改点表保存后 ≤10 秒内新点位开始采集（无需人工禁用/启用插件）。
4. **加一台新设备（新端点/新拨入连接）**同样免人工操作插件，≤30 秒内出数（验证拓扑级刷新而非仅队列替换）；删设备后对应连接停止重连。
5. 新建组态项目：数据集选"IoT 设备点位"绑定方案 A 模拟器出数的设备，运行时页面实时刷新、历史图元出曲线，**全程不手填任何 URL/token**（C-6 验收）。

### 6.4 工作量估算

设备中心骨架+树组件 4~5 天，tabs 复用改造 3~4 天，选择器化 2~3 天，点表裁剪+热刷新 3 天；C-6 组态数据源绑定 7~10 天（绑定器 UI 3~4 天、运行时 SignalR 接线 2~3 天、历史数据集+查询接口 2~3 天）。**设备中心部分约 2.5~3 周，C-6 另计 1.5~2 周（单人前端为主，C-6 可由第二人并行；热刷新需后端配合 2 天）**。
