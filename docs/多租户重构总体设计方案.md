# 标准 SaaS 多租户重构总体设计方案

> 版本：v1（待审）｜日期：2026-07-09
> 范围：`unit_id → tenant_id` 标准化 + Build/Dept 甲类清理并入 + 越权盲区收口
> 定位：物联网中台（解析 + 中转 + 多租户隔离）；空间层级（楼栋/部门）属上层业务，本次剥离
> 证据：4 轮只读探子全量扫描（实体层 / 服务鉴权层 / 控制器DTO层 / 多租户机制）

---

## 0. 一句话目标

把现有"半成品的 unit 隔离"升级为**标准 SaaS 租户隔离**：机制改名 `unit→tenant`、覆盖面补齐到全部业务实体、后台组件越权盲区收口，同时把 Build/Dept 空间层级作为"顺带清理"剥离干净——**一次规划到位，不拆了再建**。

## 1. 现状事实（重构的地基）

### 1.1 隔离机制：全局自动 + 高度集中（好消息）
| 环节 | 位置 | 作用 |
|---|---|---|
| 标记接口 | `IotModel/Common/UnitScope.cs:8` `IUnitEntity{int UnitId}` | 谁挂谁被隔离 |
| 上下文 | `UnitScope.CurrentUnitId` = `AsyncLocal<int?>` | 随请求流转，后台任务为 null 不隔离 |
| SQL 过滤 | `DbContext.cs:775` `ApplyUnitScope` → `QueryFilter.AddTableFilter<IUnitEntity>` | 查询自动加 `unit_id=当前` |
| 写入回填 | `DbContext.cs:782` `Aop.DataExecuting`（UnitId==0 才填） | 插入自动填 unit_id |
| 缓存过滤 | `DbContext.cs:798` `FilterUnitScope` | `[EntityCache]` Redis 路径出口再过滤一遍 |
| 注入 | `TokenAuthorizationFilter.cs:55` `CurrentUnitId=model.UnitId` | 每请求解密 Token 写上下文 |
| 来源 | `sys_user.unit_id` → 登录加密进 Token | 用户绑定单位 |

**结论：机制内核改动小（几个集中点改名），难在覆盖面与列名散落。**

### 1.2 覆盖面：半成品
- 仅 **12/30** 业务实体实现 `IUnitEntity`（ScadaProject/DashProject/DashDataPool/DeviceInfo/DeviceParam/DeviceComfort/DeviceAlarmConfig/DeviceTypeAlarmConfig/DeviceTypeRun/DeptInfo/BuildInfo/EventBase）。
- **6 个后台组件实体明确不隔离（越权盲区）**：PushStrategy、CollectStrategy、NorthboundSink、LinkageRule、AlarmMask、NotifyChannel——全局共享，真租户化必须逐个裁定。

### 1.3 租户表：BasicunitInfo
- 扁平结构，主键 `unit_id` 自增，无 parent_id/tree_level，当前"租户"==「单位」。

## 2. 目标架构

### 2.1 命名标准化
| 现在 | 重构后 | 说明 |
|---|---|---|
| `IUnitEntity` | `ITenantEntity` | 标记接口 |
| `int UnitId` | `int TenantId` | 属性 |
| 列 `unit_id` | 列 `tenant_id` | 物理列（需数据迁移） |
| `UnitScope.CurrentUnitId` | `TenantScope.CurrentTenantId` | 上下文 |
| `BasicunitInfo`（单位） | `TenantInfo`（租户）+ 兼容视图 | 租户主表 |

> **决策点 A**：是彻底改列名 `unit_id→tenant_id`（干净但要全量数据迁移 + 旧 Token 失效），还是**保留物理列名 `unit_id`、只改代码层属性/接口名**（零数据迁移，代码语义清晰）？后者风险低得多，见 §5 决策清单。

### 2.2 覆盖面补齐（★口径已修正 2026-07-09，探子实测）
> 原"18 个"是删 BuildInfo/DeptInfo 之前的旧口径，且混入了 DTO。**实测当前"有 UnitId 属性但未挂 IUnitEntity 的持久化实体只有 7 个"**：
> - **不纳入（数据源/租户表本身）**：Sysuser（unit_id 是隔离数据源→Token）、BasicunitInfo（租户表本身，挂了用户查不到自己单位）。
> - **归 6 盲区平台级**：PushStrategy、CollectStrategy（已有 unit_id 列预留）。
> - **P2 普通纳入候选（需业务确认是否单位级）**：SysRelated（用户单位权限映射）、AdminMqttparam（mqtt 配置）、AdminLogoparam（logo 配置）。
> - **勿误纳的 DTO（非实体）**：OperatorModel/UserInfo/MqttWebModel/EnergyAnalysis/DataChartInfo/SysuserOutput/DeviceMonitorInfo。
- 6 个后台组件盲区：定义"平台级实体"概念（`tenant_id=0` 表示平台共享）。PushStrategy/CollectStrategy 已有 unit_id 列，另 4 个（NorthboundSink/LinkageRule/AlarmMask/NotifyChannel）无 UnitId 字段，参与需先加列。

### 2.3 Build/Dept 甲类剥离（并入本次）
按 `docs/Build-Dept删除方案.md` 甲类清单执行：删 Build/Dept 管理功能全套 + 摘除事件/设备记录里的 build/dept 打标冗余列。乙类鉴权隔离逻辑（OperatorCommon/CustomActionFilter 的按建筑/部门过滤）**直接废弃**——因为标准租户模型下，隔离维度是 tenant 而非 build/dept，这套 build/dept 行级过滤本就该退场。

**Excel 设备导入处置（决策 2026-07-09：删建树、保导入）**：DeviceInfoController 的 Excel 导入现在会读 BuildId1-4/DeptId1-3 七列自动建楼栋/组织树。剥离后：
- 删「建筑段(472-617)」「部门段(619-731)」整段建树逻辑 + GetBuild/GetDepart 辅助方法。
- 保留「设备段(733-849)」设备导入主流程，去掉 GetBuild/GetDepart 调用、build/dept 存在性校验、BuildId/DeptId 赋值、ParentName 的 BuildId 匹配条件。
- 保留「单位段(444-470)」——unit 是租户维度。
- Excel 模板不再需要楼栋/部门列，前端导入模板 P4 同步调整。

### 3.1 P1~P3 终局交付实录（2026-07-12，整库重建方式）

**重大发现（交付前）**：原"隔离已由 DbContext 接管"的注释是空头支票——TenantScope 只写不读，全仓无 AddTableFilter/插入回填，隔离实际处于断路状态。本次交付的核心即真正接线。

- **租户表**：`BasicunitInfo(basicunit_info)` → `TenantInfo(tenant_info)`，极简字段：TenantId(tenant_id,PK自增) + 树5字段(parent_id/tree_level/full_code/full_name/has_child) + tenant_name + remark。删除：UnitCode/AreaId/AreaName/StructureArea/UsableArea/TotalPeople/UnitAddress/ExpandJson(EnergyRate 全仓零消费、RouterPath 前端不读)。区域绑定连带删 SysAreaController.GetSunTimes 及 SunTimesCalculator/SunTimesResult 孤儿文件。
- **隔离内核**（IotModel/Common/TenantScope.cs）：TenantScope(AsyncLocal 上下文) + `TenantIsolation.Attach`，挂载点三处——`DbContext<T>.GetOperDb`、`TranAction`、`SqlSugar_Db` configAction（CopyNew 不继承过滤器，必须逐实例挂）。过滤谓词：超管/无上下文豁免，否则 `tenant_id=0(平台共享,决策B) OR IN 可见集(父见子孙,决策C)`；插入回填：非超管且 TenantId==0 时盖当前租户，超管保留显式值(0=平台级)。
- **缓存路径**：EntityCache 全表加载加 `ClearFilter()`（否则先到租户会把自己的子集写成"全表"缓存），六个缓存分支出口统一走 `FilterTenantScope` 内存过滤。ValidateCacheConsistency 对账同样 ClearFilter。
- **已知不覆盖面**：Fastest/BulkCopy 与 Ado 原生 SQL 不经过滤器与回填，调用方须显式赋 TenantId（事件入库 DataPointIngestService 已显式赋值）。
