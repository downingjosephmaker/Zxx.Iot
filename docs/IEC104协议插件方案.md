# IEC 60870-5-104 协议插件方案

> 目标：新建 `IotPlugin.Iec104`，以**主站**身份接入变电站/RTU 子站。
> 依赖：[协议通用工具库与脚本注入方案.md](./协议通用工具库与脚本注入方案.md) 的 `ValueCodec`（仅此一项）。

## 一、先说清楚：IEC104 和现有插件不是同一个物种

### 1.1 它从通用工具库里能复用的，非常少

| 工具 | 645 / 188 | IEC104 |
|---|---|---|
| CRC16 / 累加和 | ✅ | ❌ **IEC104 没有校验码**，完整性交给 TCP |
| BCD | ✅ | ❌ 纯二进制 |
| `FE` 唤醒前导 | ✅ | ❌ 没有 |
| 字节序 / float32 解码 | 部分 | ✅ **唯一真用得上的**（TI=13 短浮点数，IEEE754 小端） |
| 进制转换 | 调试 | 调试 |

**不要为了"看起来统一"把 IEC104 塞进 645/188 的模子。** 它需要的是别的东西。

### 1.2 真正的难点：轮询问答 vs 长连接上报

- **645 / 188 / Modbus** = 轮询问答。框架合包 → 驱动发帧 → 等应答 → 返回。`IDeviceDriver.ReadBatchAsync(PointBatch)` 就是为它们量身定做的。
- **IEC104** = 长连接主动上报。连上 2404 → `STARTDT_act` → 一次**总召唤**拿全量 → **之后子站变化时自己推过来**。压根没有"读第 5 到第 10 个点"这回事。

外加 645/188 完全没有的：I/S/U 三种帧格式、收发序号确认窗口（k/w）、四个超时定时器（t0~t3）、CP56Time2a 七字节时标。

## 二、架构：插件自己扛，不动核心契约

```
         ┌─ 接收循环（后台任务）
TCP 长连接 │    ├─ U帧 → 状态机（STARTDT/TESTFR 应答）
 :2404    │    ├─ S帧 → 更新已确认序号
         │    └─ I帧 → 解 ASDU → 刷新缓存
         └──────────────┬──────────────
                        ↓
              最新值缓存  (CA, IOA) → {value, quality, ts}
                        ↑
   ReadBatchAsync(batch)─┘   不发帧，直接返回缓存快照
```

**`IDeviceDriver` 一个字都不改。** 框架的"采集周期"语义自然变成"快照周期"。

### 为什么不给 `IDeviceDriver` 加 push 语义

Modbus / 645 / 188 / S7 / OpcUa **全都用不上**。为一个插件改动所有插件共同依赖的核心契约，收益与风险严重不匹配。缓存快照是工业界处理这类"上报型协议接入轮询型框架"的标准做法。

### 通道复用

IEC104 是纯 TCP 长连接客户端，`IotDriverCore/Channels/TcpClientChannelPool` 已具备连接维持与重连退避，直接复用。
它**不涉及** UDP / MQTT / HTTP —— 那是正交的另一件事。

## 三、点表映射：`DriverPoint` 几乎原样可用

| `DriverPoint` 字段 | IEC104 语义 |
|---|---|
| `SlaveAddr` | **公共地址 CA**（ASDU 公共地址，2 字节） |
| `Address` | **信息体地址 IOA**（3 字节，1 ~ 16777215） |
| `FuncCode` | **类型标识 TI**（1 字节，天然对位） |
| `DataType` | 交给 `ValueCodec` |
| `ByteOrder` | 短浮点固定 `DCBA`（IEC104 小端） |
| `BitOffset` | 单点/双点信息的位提取 |
| `Scale` | 归一化值满量程换算 |
| `DriverPointValue.Quality` | **QDS 品质描述词直接映射**（IV/NT/SB/BL/OV → 非 0） |

**点表模型零改动。** `PointBatch` 的合包对 IEC104 无意义，但也无害——按 `batch.Points` 逐个从缓存取值即可，是一次无成本空转。

## 四、协议要点（实现依据）

### 4.1 APCI — 6 字节

```
0x68 | 长度(1B, ≤253) | 控制域(4B)
```

由控制域首字节低 2 位分派帧格式：

| 格式 | 判据 | 含义 |
|---|---|---|
| **I** | bit0 = 0 | 信息传输。含发送序号 N(S) 与接收序号 N(R)，各 15 bit |
| **S** | 低2位 = `01` | 编号监视。仅 N(R)，用于确认 |
| **U** | 低2位 = `11` | 未编号控制 |

U 帧控制域首字节取值：

| 报文 | act | con |
|---|---|---|
| STARTDT | `0x07` | `0x0B` |
| STOPDT | `0x13` | `0x23` |
| TESTFR | `0x43` | `0x83` |

### 4.2 ASDU

```
TI(1B) | VSQ(1B) | COT(2B) | CA(2B) | 信息体…
```

- **VSQ**：bit7 = SQ 顺序位，bit0-6 = 信息体数目
  - `SQ=0`：每个信息元素各带自己的 IOA
  - `SQ=1`：只有首个 IOA，后续地址**连续递增**（解析时最易出错处）
- **COT**：bit0-5 传送原因，bit6 = P/N，bit7 = T(测试)；第 2 字节 = 源发站地址
- **IOA**：3 字节

常用 COT：`3`=突发，`5`=请求，`6`=激活，`7`=激活确认，`10`=激活终止，`20`=响应站召唤。

### 4.3 第一版支持的类型标识

**监视方向（上行）**

| TI | 名称 | 说明 |
|---|---|---|
| 1 | M_SP_NA_1 | 单点信息 |
| 3 | M_DP_NA_1 | 双点信息 |
| 9 | M_ME_NA_1 | 测量值，归一化值 |
| 11 | M_ME_NB_1 | 测量值，标度化值 |
| 13 | M_ME_NC_1 | 测量值，短浮点数 |
| 30 / 31 / 34 / 36 | *_TB_1 / *_TD_1 / *_TF_1 | 上述四种的 CP56Time2a 带时标版 |

**控制方向（下行）**

| TI | 名称 |
|---|---|
| 45 | C_SC_NA_1 单点命令 |
| 46 | C_DC_NA_1 双点命令 |
| 50 | C_SE_NC_1 设定值命令，短浮点数 |

**系统**

| TI | 名称 |
|---|---|
| 100 | C_IC_NA_1 总召唤 |
| 103 | C_CS_NA_1 时钟同步 |

**第一版不做**：累计量（15/16）、计数量召唤（101）、参数设定（110+）、文件传输（120+）。等有真实现场诉求再加。

### 4.4 时序参数（可配置，默认值取自标准）

| 参数 | 默认 | 含义 |
|---|---|---|
| k | 12 | 发送方未被确认的 I 帧上限，到顶即停发 |
| w | 8 | 接收方每收 w 个 I 帧必须发 S 帧确认 |
| t0 | 30s | 连接建立超时 |
| t1 | 15s | 发送 APDU 后等确认超时 |
| t2 | 10s | 无数据时发确认超时（须 t2 < t1） |
| t3 | 20s | 长期空闲发 TESTFR 超时 |

### 4.5 启动时序

```
TCP连接 → STARTDT_act → STARTDT_con
       → C_IC_NA_1 (COT=6 激活, QOI=20 站召唤)
       → COT=7 激活确认
       → I帧 × N (COT=20 响应站召唤，全量点)
       → COT=10 激活终止
       → 之后子站变化时主动上报 (COT=3 突发)
```

### 4.6 遥控：选择后执行（SBO）

命令帧的 SCO/DCO 限定词含 S/E 位：

- **直接执行**：S/E=0，一步下发
- **选择后执行**：先发 S/E=1 选择 → 等激活确认 → 再发 S/E=0 执行

**电力行业遥控普遍强制 SBO。** 插件须两种都支持，按配置项选择，默认 SBO。

## 五、插件骨架（遵循本仓既有插件惯例）

```
IotPlugin.Iec104/
├── Iec104Plugin.cs          # IDeviceDriver 实现 + 接收循环 + 缓存
├── Iec104PluginConfig.cs    # [ConfigParameter] 标注，平台反射生成表单
├── Iec104FrameHelper.cs     # APCI/ASDU 编解码
├── Iec104StateMachine.cs    # STARTDT/序号窗口/t0~t3 定时器
├── Cp56Time2a.cs            # 七字节时标编解码（IEC 系列专属，不进公共库）
└── Iec104ControlCommand.cs  # ContentJson 反序列化模型
```

类上标注 `[DriverInfo("IEC104", "1.0.0", "...")]`，配置属性标注 `[ConfigParameter(...)]` —— 平台反射生成前端表单，**新驱动零 UI 代码**。

`Cp56Time2a` 刻意留在插件内：它是 IEC 60870-5 系列专属概念，放进 `IotDriverCore/Codec/` 会污染协议层。

## 六、模拟器：第二阶段再做

`ISimulatable` 是**可选**契约——`IotPlugin.OpcUa` 与 `IotPlugin.S7` 都没有实现它。

IEC104 第一版**不实现** `ISimulatable`，理由：模拟一个 104 从站等于把 I/S/U 状态机、序号窗口、总召唤应答**再实现一遍对侧**，工作量与主站相当，会把第一版拖成两倍长。

但必须承认：**IEC104 恰恰是最需要模拟器的协议**——现场很难借到真实 RTU 做联调。所以它是明确的第二阶段项，不是"不做"。

届时若做，须遵守本仓铁律：**从站编解码独立实现，不与插件主体共用代码**，否则对抗性验证退化为自证。

## 七、风险

| 风险 | 缓解 |
|---|---|
| **`SQ=1` 连续地址解析** —— 最常见的实现 bug，会导致点位整体错位 | 单测覆盖 `SQ=0` 与 `SQ=1` 两条路径，用真实报文样本 |
| **序号窗口 k/w 处理错误** —— 表现为跑一段时间后子站单方面断链 | 单测覆盖窗口到顶停发、收 w 帧发 S 帧 |
| **缓存陈旧** —— 子站断链后缓存仍返回旧值，平台误以为在线 | 断链即将缓存全部标记 `Quality` 非 0；重连后重发总召唤刷新 |
| **ALC 类型分裂** —— 新插件引用了改动后的 `IotDriverCore` | 见工具库方案 §5.1：所有在线插件包必须**全部**重新编译上传 |
| **CP56Time2a 时区/夏令时** | 第一版按子站本地时间原样解析，不做时区换算；如现场有诉求再议 |

## 八、验证

1. **单测优先**（无需真实设备）：
   - APCI 三种帧格式编解码往返
   - ASDU 各 TI 解码，用真实报文样本
   - `SQ=0` / `SQ=1` 两条路径
   - 序号窗口：k 到顶停发、每 w 帧发 S 帧
   - CP56Time2a 编解码往返
2. **状态机测试**：STARTDT 握手、t1 超时重连、t3 空闲发 TESTFR
3. **端到端**：待第二阶段模拟器就绪，或现场借到 RTU 后补做

## 九、与工具库方案的关系

本方案**只依赖**工具库方案的一项：`ValueCodec.Decode(hex, "float32", "DCBA", -1)` 用于 TI=13/36 短浮点数。

其余（CRC、BCD、`FE` 前导、累加和）IEC104 全都用不上。
因此**两个方案可以并行推进**，工具库方案不必等 IEC104，IEC104 也只需等 `ValueCodec` 提升完成。
