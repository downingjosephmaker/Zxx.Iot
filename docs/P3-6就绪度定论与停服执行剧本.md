## 2. 核心判断(为何可以放心停服)

1. **五路侦察 + 对抗验证一致收敛**:全部环节代码侧 DONE,无一处落到"代码缺失"。蕾姆亲读 `DbContext.cs` 与 `UnitScope.cs` 复核,证据字字属实。
2. **所有"遗留"都不是代码缺陷,而是数据侧/设计约束**:存量单位经 V0002 后全为扁平根租户(`full_code=|unit_id|`、`parent_id=0`),此状态下可见集退化为"只见自身"。这是**功能生效的数据前提**,不是代码是否写完的问题。
3. **安全侧零风险**:平级存量下可见集退化只会"看不到"、绝不"越权看到"(`UnitScope.cs:85-86` 兜底);超管靠 `CurrentIsSystem` 全局豁免(`DbContext.cs:781/810`),不依赖可见集。最坏情况 = 退化为旧单租户隔离行为,不会数据越权泄露。

## 3. 若现在停服一起做会踩的坑(按严重度)

> **必须区分两件事**:跑 V0002 加字段本身安全(纯新增列 + 存量回填);"父见子孙"能力要等**建树**(有人经 SaveBatch 选父级)才生效,**不是等代码补齐——代码已齐**。

| 级别 | 坑 | 应对 |
|---|---|---|
| 🔴 **P0-硬门槛** | Migration 开关默认 true → CodeFirst 会抢跑,与手写 V0001/V0002 冲突,按实体 IsNullable 生成 nullable 列漂移 V0002 的 NOT NULL 设计 | **停服前先关 Migration 开关**(`Config/DbSetting.config`),或确保手写脚本先落地 |
| 🟠 **P1-业务预期** | "停服恢复后父租户立刻能看到子租户数据"的预期在建树前不成立(所有非超管仍只见自身) | **提前告知昴君:迁移完成 ≠ 父见子孙生效,还差建树一步** |
| 🟠 **P1-授权语义** | 一旦建树,父租户可见范围立即扩大到全部子孙——B1"父见子孙"须最终确认 | 停服前确认 B1 确为期望授权语义 |
| 🟡 **P2-数据完整性** | 级联删/级联重算强依赖存量 `full_code` 正确回填 | 停服脚本校验 V0002 回填结果:`full_code` 非空、格式 `|unit_id|` |
| 🟡 **P2-隔离暴露面** | 旁路入口(未标 `[Token]` 的端点、`GetIotTripartite` 三方入口)不装配 TenantScope → `CurrentTenantId=null` 放行全表 | **既有设计属性,非本次回归**;停服上线是审视"涉租户端点是否都带 `[Token]`"的时机 |
| ⚪ **P3-事务中间态** | ① SaveBatch 批内无跨条原子性;② DAO Update override 无显式事务;③ SaveBatch 依赖同批 list 父在子前(无重排) | 建树时保证传入顺序父在子前;批量失败后人工核对中间态 |
