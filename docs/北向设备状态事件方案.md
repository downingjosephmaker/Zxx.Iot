# 北向设备状态事件方案

> 目标：让第三方能收到**可靠的设备上线/离线事件**，而不是从告警流里靠字符串推断。
> 定位：**北向**（数据出去）。与南向接入是反方向，见 [南向TCP-UDP上报接入方案.md](./南向TCP-UDP上报接入方案.md)。

## 一、现状：北向只有两个出口

`IotWebApi/Services/NorthboundForwardService.cs`（§10.2）功能完整——内存队列 + 落盘重传 + 多目的地 + 状态查询 + 测试发送。**不是空壳**。

但它对外**只开了两个入口**，全仓调用点已查尽，一个不多：

```
ForwardTelemetry(deviceid, typecode, points)   ← 仅 DataPointIngestService:354   = 上报
ForwardAlarm(deviceid, typecode, signal)       ← 仅 DataPointIngestService:694   = 告警
```

**没有第三个。设备上线/离线没有独立的北向事件。**

## 二、离线/上线现在是怎么出去的

以**告警**的形态：

- 离线 → `AlarmEventType = "离线"` 的告警（`AlarmEngineService.FireOffline`）
- 上线 → 同类告警，靠 `Content` 是 `"[恢复]设备通信恢复上线"` 来区分（`AlarmEngineService.FireOfflineRecover`）

第三方收到的是**一条告警**，不是**一个状态事件**。要判断设备在不在线，得先从告警流里滤出 `AlarmEventType == "离线"`，再看 `Content` 开头有没有 `[恢复]`。

## 三、三个坑：为什么这条路不可靠

| # | 坑 | 位置 | 后果 |
|---|---|---|---|
| 1 | **依赖字典配置** | `AlarmEngineService:299-301` | 离线告警字典行靠"AlarmType 或 EventType 含'离线'字样"认定。没配就走默认（不通知） |
| 2 | **🚨 屏蔽会吃掉它** | `AlarmLifecycleService:29` | "完全屏蔽的告警不进生命周期"。运维一屏蔽离线告警，**第三方从此再收不到任何离线通知** |
| 3 | **🚨 边沿检测** | `AlarmEngineService:628-635`，`_offlineActive` | 只对**发过离线告警**的设备发恢复。设备**首次上线永远没有事件** |

根子是同一个：

> **设备状态是客观事实，告警是运维策略。把事实建在策略上面，策略一改，事实就没了。**

## 四、好消息：判定服务本来就是独立的

`OfflineDebounceService`（§7.5）是一个干净的 `IHostedService`，**用委托对外解耦**：

```csharp
public Func<int> ConfirmSecondsProvider { get; set; }   // 可选。未注册回落 ConfirmSeconds=180 兜底
public Action<List<(int,int,string)>> ConfirmHandler    // 离线确认出口
public void OnSuspectOffline(int deviceid, int devicestate)
public void CancelSuspect(int deviceid)
```

关键：**它不依赖告警**。`ConfirmSecondsProvider` 只是个可选委托——当前由告警引擎注册以读 `AlarmConfirmSeconds`，但**未注册时回落 180 秒兜底**（`OfflineDebounceService.cs:20`），判定照跑。

判定逻辑本身是独立的：离线先进疑似中间态，持续超确认时长才正式判离线，期间恢复则静默取消；上线即时生效，通知 60 秒限频防抖。

**所以我们不需要"绕开"任何东西——只需要在既有的判定结果上，并联一个出口。**

而且设计者本来就是这么想的。`DataPointIngestService:543` 的注释白纸黑字写着：

> "离线告警**与上下线事件**共用同一判定结果(§9.6:确认时长即时长型防抖…)"

**"上下线事件"这个概念一直在，只是它的北向出口从来没建。** 本方案就是把这个出口补上。

## 五、设计：并联第三个出口

```
OfflineDebounceService（已独立判定，不动它）
   │
   ├─→ AlarmEngineService.FireOffline / FireOfflineRecover   （现有：运维策略，可被屏蔽）
   │
   └─→ NorthboundForwardService.ForwardDeviceState           （新增：客观事实，不可屏蔽）
```

### 5.1 新增入口

```csharp
// NorthboundForwardService
public void ForwardDeviceState(int deviceid, string typecode, int devicestate, string reason)
```

与既有 `ForwardTelemetry` / `ForwardAlarm` 同构：入队即返回，复用同一套队列、落盘重传、多目的地分发。

### 5.2 状态语义（沿用平台既有定义，不新造）

| 值 | 含义 | 来源 |
|---|---|---|
| `0` | 离线 | `OfflineDebounceService.SuspectState.DeviceState` 注释 |
| `1` | 掉电 | 同上 |
| `2` | 在线 | `ProtocolScriptService.BuildDeviceData` 中 `dev.DeviceState = 2` |

**注意有「掉电」这一态**，不是简单的上线/离线二值。北向报文须如实带出三态。

### 5.3 接线：精确到两行

| 位置 | 现有代码 | 并联 |
|---|---|---|
| `DataPointIngestService:477` | `_alarmEngineService.FireOfflineRecover(data.DeviceId)` | `_northboundForwardService.ForwardDeviceState(data.DeviceId, typecode, 2, "通信恢复")` |
| `DataPointIngestService:547` | `_alarmEngineService.FireOffline(fire.DeviceId, fire.Reason)` | `_northboundForwardService.ForwardDeviceState(fire.DeviceId, typecode, fire.DeviceState, fire.Reason)` |

`DataPointIngestService:106` **已经注入了** `_northboundForwardService`，无需改构造。

### 5.4 三个坑如何被解掉

| 坑 | 解法 |
|---|---|
| 1. 依赖字典配置 | 新出口不查告警字典。判定服务本就有 180 秒兜底 |
| 2. 屏蔽会吃掉 | 新出口不进告警生命周期，屏蔽规则管不着它 |
| 3. 首次上线无事件 | 新出口不看 `_offlineActive` 边沿标记，上线判定成立即发 |

## 六、报文形态

与既有两类保持同构（具体字段随 Sink 的现有序列化约定，不另造格式）：

```
{ 事件类型: "deviceState", 设备ID, 产品类型编码, 状态: 0|1|2, 原因, 时刻 }
```

第三方据此可直接维护设备在线表，**不必再解析告警文本**。

## 七、验证

1. **单测**：`ForwardDeviceState` 入队与分发；三态取值正确传出。
2. **集成**：
   - 设备停止上报 → 超过确认时长 → 第三方收到 `状态=0`
   - 设备恢复上报 → 第三方**立即**收到 `状态=2`
   - **屏蔽离线告警后**，上述两条**仍然成立**（本方案的核心断言 —— 坑 2 的回归）
   - **从未离线过的设备首次上线** → 第三方收到 `状态=2`（坑 3 的回归）
3. **回归**：现有离线告警与恢复行为**逐字不变**——本方案只并联，不改动告警链路任何一行。

## 八、不做什么（YAGNI）

- **不动 `OfflineDebounceService`**。它的判定逻辑是对的，且已用委托解耦，无需改造。
- **不动告警侧的离线告警/恢复**。运维要的是告警，第三方要的是事件，两者并存互不干扰。删掉告警侧会破坏现有运维流程。
- **不做状态事件的历史落库**。北向转发是"发出去"，落库是另一件事；平台侧设备状态已在 `DeviceInfo.DeviceState` 有最新值。
- **不做网关连带状态合并**。`OfflineDebounceService.cs:10` 注释已标注"网关连带合并待 M3 通道/网关拓扑落地后接入"——那是既有的独立议题，不在本方案范围。
